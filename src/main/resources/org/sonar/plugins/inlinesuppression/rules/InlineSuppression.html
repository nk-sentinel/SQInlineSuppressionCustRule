Inline suppression of static analysis findings is not allowed

Using inline suppression mechanisms to bypass SonarQube static analysis is a security risk. Suppressions hide potential vulnerabilities, bugs, and code smells from review, undermining the quality gate and compliance requirements.

This rule detects the following suppression patterns:

Pattern Languages Description
// NOSONAR All Suppresses all SonarQube rules on the annotated line. Also detected in /* NOSONAR */, # NOSONAR, <!-- NOSONAR -->, and -- NOSONAR comment styles.
@SuppressWarnings("java:S1234") Java, Kotlin Suppresses specific SonarQube rule(s). Detected for all language rule prefixes (java:, squid:, csharpsquid:, javascript:, typescript:, python:, kotlin:, go:, php:, ruby:, scala:, vbnet:, xml:, css:, web:, plsql:, tsql:, c:, cpp:, and more). Also detected: @SuppressWarnings("all") and multi-rule suppression @SuppressWarnings({"java:S106", "java:S1186"}).
@Suppress("kotlin:S1234") Kotlin Kotlin-specific annotation for suppressing SonarQube rules.
[SuppressMessage("SonarAnalyzer", "S1234")] C#, VB.NET Suppresses SonarQube Roslyn analyzer rules. Detected with optional fully-qualified name [System.Diagnostics.CodeAnalysis.SuppressMessage(...)] and attribute targets (assembly:, module:, etc.).

Noncompliant code examples

Java — NOSONAR

System.out.println("debug"); // NOSONAR — Noncompliant

Java — @SuppressWarnings (single rule)

@SuppressWarnings("java:S106")  // Noncompliant
public void log() {
    System.out.println("message");
}

Java — @SuppressWarnings (multiple rules)

@SuppressWarnings({"java:S106", "java:S1186"})  // Noncompliant
public void log() {
    System.out.println("message");
}

Java — @SuppressWarnings("all")

@SuppressWarnings("all")  // Noncompliant
public class MyClass { }

Kotlin — @Suppress

@Suppress("kotlin:S1234")  // Noncompliant
fun process() { }

C# — [SuppressMessage]

[SuppressMessage("SonarAnalyzer", "S1234")]  // Noncompliant
public void Process() { }

Python — NOSONAR

x = eval(input)  # NOSONAR — Noncompliant

Compliant solution

Fix the underlying issue reported by SonarQube instead of suppressing it:

// Instead of suppressing S106, use a proper logger:
private static final Logger LOG = LoggerFactory.getLogger(MyClass.class);

public void log() {
    LOG.info("message");  // Compliant
}

Exceptions

This rule does not flag @SuppressWarnings annotations that only suppress standard Java compiler warnings (e.g., "unchecked", "deprecation", "serial") without any SonarQube rule references.

Future improvements

  • #pragma warning disable S1234 (C#) — pragma-based suppression
  • @Generated / @javax.annotation.Generated (Java) — generated code exclusion
  • [GeneratedCode] (C#) — generated code attribute exclusion
